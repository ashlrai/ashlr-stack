v0.1 shipped three weeks ago with a broken promise. The CLI installed fine.
The template scaffolded. But `stack init --template nextjs-supabase-posthog`
didn't provision Supabase — it printed a placeholder. The SDKs didn't install
— you still had to run `npm install` separately. And if you tried to install
via npm, you hit our 2FA wall immediately, because our release automation
wasn't handling OTP entry. The "one command for your whole stack" headline
was technically true in the code; it just didn't work in practice.

v0.2 fixes all three.

## What was broken and what we did about it

**Problem 1: Template didn't provision.** `stack init --template X` created
the file structure but treated provisioning as a separate step. That's not
"one command" — that's just a scaffolder with extra opinions.

Fix: `stack init --template nextjs-supabase-posthog` now runs the full
provisioning pipeline inline. Browser opens, Supabase OAuth dance completes,
project is created, three secrets land in Phantom, `.env.local` gets patched.
All inside the init flow.

**Problem 2: No SDK install.** v0.1 wired the secrets and the MCP config, but
left you to `npm install @supabase/supabase-js` yourself. Which sounds minor
until you realize you now have to know the right package name for each
provider.

Fix: v0.2 prompts inline — "install @supabase/supabase-js? (Y/n)" — and runs
the install via your detected package manager (bun, pnpm, npm). The prompt is
skippable if you want to manage deps yourself.

**Problem 3: npm 2FA blocker.** We publish to npm, and npm requires
interactive OTP entry for scoped packages. Our CI can't do that unattended,
which meant the npm package lagged or silently failed.

Fix: we ship a Homebrew tap now. `brew install ashlrai/ashlr/stack` is the
recommended install path. The npm package still exists for environments where
brew isn't available, but if you can use brew, use brew.

## What's new beyond the fixes

**swap + rollback.** The feature that makes Stack feel like a control plane
rather than a provisioner:

```bash
stack swap supabase --to neon
```

Migrates a live project from Supabase to Neon. Provisions the Neon database,
rotates the secrets in Phantom, updates every env reference in `.env.local`
and `.stack.toml`. `stack rollback supabase` if it goes wrong.

This is the thing v0.1 was missing to be useful beyond day-one setup.

**Cloudflare Worker telemetry endpoint.** Stack itself has no telemetry. v0.2
ships a slim Cloudflare Worker that counts install events — just a counter,
no project data, no secrets, no identifiers. It's in the repo, auditable.
We added it so we know whether anyone is actually running the thing.

## How the provisioning pipeline works

`stack add supabase` (and the inline version in `stack init`) does four things
in sequence:

1. **Login.** Where we have an OAuth app registered (Supabase, GitHub), we
   open the browser, run PKCE, capture the access token. For the other 34
   providers, we prompt for a PAT or API key and verify it against a
   read-only endpoint before going further. You never commit a bad key
   halfway through.
2. **Provision.** We hit the provider's API to create the upstream resource.
   Supabase: project in `us-east-1`. Neon: branch + compute. Turso: database
   + auth token. Where a provider doesn't expose a creation API (Stripe,
   Linear), we bind to an existing account.
3. **Materialize.** Every secret the provider emits goes into **Phantom** —
   an encrypted local vault you install once per machine. `.env.local` and
   `.stack.local.toml` get a `phm_…` placeholder token, not the real value.
   The raw secret never touches disk.
4. **Persist.** We patch `.stack.toml` (committed — declares which services
   this project uses), `.stack.local.toml` (gitignored — instance IDs,
   regions), and `.mcp.json` (so Claude / Cursor can reach the service's MCP
   server).

At runtime, `stack exec -- bun dev` resolves the `phm_…` placeholders to
real secrets in the child process environment. If `.env.local` leaks in a
screenshot or an accidental commit, the attacker gets meaningless tokens.

## 39 providers, same model

Every provider goes through the same four steps. The catalog covers:

- **DB:** Supabase, Neon, Turso, Convex, Firebase, PlanetScale, Xata
- **Deploy:** Vercel, Railway, Fly.io, Render, Cloudflare Workers
- **Cloud:** AWS
- **AI:** OpenAI, Anthropic, xAI, Groq, DeepSeek, Together, Fireworks
- **Obs:** Sentry, PostHog, Datadog, LogFlare
- **Auth:** Clerk, Auth0
- **Payments:** Stripe, Lemon Squeezy
- **Infra:** Upstash, Cloudflare KV, Resend, Loops
- **Dev:** GitHub, Linear, Slack

5 of 39 are full OAuth (browser-open, no manual key paste). The rest use
PAT or API-key paste with automated verification. OAuth app registrations
are in progress — a week to a month per provider depending on their review
process.

## Honest v0.2 caveats

This is a launch post, not a press release.

- **Pre-alpha.** We use Stack daily on real projects. We wouldn't bet a team
  of 50 on it yet. The data model is stable; the CLI surface will add
  commands but won't remove them until 1.0.
- **npm 2FA blocker is not fully resolved.** The release path is improved,
  but interactive OTP is still required for manual npm publishes. Use brew.
- **Supabase OAuth app — dashboard-only limitation.** Supabase can't
  programmatically create orgs; only projects within existing orgs. Fresh
  Supabase accounts need one manual dashboard action first. We're in
  conversation with Supabase about a workaround.
- **Phantom is required.** Not optional. One install per machine.
- **Telemetry:** the CLI itself has none. The install-count Worker is opt-out
  (it fires on `stack init`; `--no-telemetry` suppresses it).

## Roadmap

1. Finish OAuth app registrations — biggest remaining papercut.
2. End-to-end recorded tests per provider — the bar for "battle-tested".
3. Better error messages — each non-trivial error rewritten as _what happened
   · why · what to do_.
4. Phantom Cloud team sync — shared vaults, RBAC, audit log.
5. More providers — Modal, Replicate, Segment, Mailchimp are top requests.
6. Windows — WSL2 works today; native Bun on Windows is close.
